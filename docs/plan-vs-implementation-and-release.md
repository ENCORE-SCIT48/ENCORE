# 계획 vs 구현 총점검 · 머지 후 발표/시연 준비

**팀 3명 기준**으로, 계획 대비 구현 현황 → 머지 → 최종점검 → PPT·현장 시연 준비까지 한 번에 정리.

---

## 1. 계획 대비 구현 현황 (총점검)

### 1.1 구현된 화면·API (계획 대응)

| 구분 | 계획(명세/테스트케이스) | 구현 여부 | 비고 |
|------|-------------------------|-----------|------|
| **인증** | 회원가입/로그인 | ✅ | `/auth/join`, `/auth/login` |
| **프로필** | 프로필 선택·전환, 유저/공연자/호스트 설정·조회 | ✅ | `/profiles/select`, `/userprofile`, `/performerprofile/*`, `/hostprofile/*` |
| **피드** | /feed, 찜한 공연 30분 전 알림 | ✅ | `/feed`, performance_schedule 연동 |
| **공연** | 목록/상세/리뷰(공연·좌석)/찜/본공연 | ✅ | `/performances`, 리뷰·좌석리뷰 분리 |
| **공연장** | 목록/상세/등록·수정·삭제/폼/내 공연장/대관 신청·내 대관·승인·거절 | ✅ | `/venues`, `/venues/my`, `/venues/{id}/reservation`, `/reservations/*` |
| **채팅** | 공연별 채팅방 목록·글 작성·상세·수정·삭제·참가·메시지·나가기·참가자 | ✅ | `/performances/{id}/chats`, `/chat/{roomId}` |

| **공연자** | 공연자 모집글·공연 모집글·신청·승인·거절·추천·내 글/신청/모집/대관/관리 | ✅ | performerPost, performancePost, manage, appliedPosts 등 |

### 1.2 정체성·추가 기능

- **Encore pick**: ✅ 구현됨. 공연 리뷰 작성 시 "이 공연에서 가장 기억에 남는 곡/장면" 한 줄(선택) 입력 → 상세 리뷰 카드·모달에 "🎵 Encore pick · ..." 노출.
- **슬로건**: **"무대를 잇다 ENCORE"** — 첫 화면/피드 상단에 노출 예정, **아직 미구현** (추후 적용).
- **오늘의 공연**: 노출 위치 → **피드(`/feed`) 상단** 또는 **공연 목록(`/performances`) 상단** 중 한 곳에 "오늘 날짜 공연" 블록으로 표시 예정. (구현 시 `performance_schedule` 기준 오늘 `startTime` 조회.)

### 1.3 의도적으로 미구현/추후

- **공연장 리뷰**: 미도입
- **관리자(어드민)**: 추후 개발
- **긴급 매칭**: 미구현
- **세트리스트**: 없음 (공연장 미확정 등록만 가능)
- **채팅 유형 태그**: 미반영

### 1.4 테스트 케이스 기준 알려진 이슈 2건

| ID | 내용 | 현재 | 조치 |
|----|------|------|------|
| **V-C-10-14** | 채팅방 나가기 → 채팅방 목록으로 이동 | 나가기 성공 후 `location.href = '/chats'` (chatJoinList) | 테스트는 `/chat/list` 기대. **동작은 됨** → URL만 문서 통일하거나, 이동이 안 되던 이슈면 `/chats`로 리다이렉트 확인 |
| **V-U-4-2** | 알림 설정 변경 후 새로고침 시 유지 | API: GET/PUT `/api/users/settings/notifications` 존재. **저장 주체(계정 vs 프로필)** 미정으로 F | DB/프로필 정책 정한 뒤 저장·조회 일치시키기 |

## 2. 머지 후 할 일 (체크리스트)

- [ ] 브랜치 통합 후 `./gradlew build` 성공
- [ ] `data-test-insert.sql` 실행해 핵심 시나리오(로그인·피드·공연·공연장·채팅·DM·대관) 한 번씩 수동 확인
- [ ] 알려진 Err 2건: V-C-10-14(나가기 이동), V-U-4-2(알림 유지) — 수정할지/발표에서 제외할지 결정
- [ ] 발표/시연용 **테스트 계정·데이터** 고정 (예: user1@test.com / performer1 / host1, 비밀번호 동일)

## 3. 발표용 PPT 구성 제안

1. **프로젝트 소개** (1장): 서비스 한 줄, 타깃(관람객/공연자/공연장 제공자)
2. **기능 개요** (1~2장): 역할별 핵심 기능 표
3. **시스템 구조** (1장): 백엔드(Spring)·프론트(Thymeleaf+JS)·DB
4. **주요 화면** (3~5장): 스크린샷 + 한 줄 설명 (피드, 공연, 공연장, 채팅, DM, 마이페이지, 대관)
5. **기술 스택·역할 분담** (1장): 3명 역할
6. **한계·추후 계획** (1장): 미구현(관리자, 긴급 매칭 등)·알림 저장 이슈
7. **데모 안내** (1장): 다음 단계에서 보여줄 시나리오

---

## 4. 현장 시연 준비 (3명 역할 분담 제안)

### A담당: 환경·데모 시나리오 정리

- 발표 PC에서 **실행 환경** 통일 (Java, DB, 포트, 테스트 데이터 적용)
- **시연 시나리오** 1개 문서로 정리 (예: 로그인 → 피드 → 공연 상세 → 찜 → 채팅방 참가 → DM)
- 예비 시나리오 1개 (네트워크/오류 대비): 예) 공연장 목록 → 대관 신청 → 호스트 승인

### B담당: PPT + 발표 흐름

- 위 PPT 구성으로 슬라이드 작성
- 발표 스크립트(1~2분/장) 및 “여기서 데모로 넘어갑니다” 구간 표시
- 데모 구간은 A담당과 연결

### C담당: 최종 점검·체크리스트

- 머지 후 체크리스트 실행 (빌드, 테스트 데이터, 핵심 경로)
- 테스트 케이스 중 **시연에 사용할 항목**만 골라서 한 번씩 실행 (OK/Err 메모)
- 발표 당일 **체크리스트·롤백 순서** 정리 (문제 시 “데모 스크립트 2로 전환” 등)

## 5. 시연 시나리오 예시 (데모용)

**시나리오 1 (약 3~5분)**  
1. 로그인 (관람객 계정)  
2. 피드에서 “찜한 공연 30분 전” 또는 공연 카드 확인  
3. 공연 상세 → 찜하기 / 공연 리뷰 또는 좌석 리뷰  
4. 공연별 채팅방 목록 → 채팅방 참가 → 메시지 전송  
5. DM 목록 → DM 방 입장 또는 수락/거절  
6. 마이페이지 → “찜한 공연” / “내가 본 공연” 이동  

**시나리오 2 (공연자/호스트, 약 2~3분)**  
1. 공연자로 전환 → 공연자 모집글 또는 공연 모집글 작성/목록  
2. 호스트로 전환 → 공연장 등록(또는 내 공연장) → 대관 관리에서 승인/거절  
3. 공연자로 → 대관 신청 → 내 대관 목록에서 상태 확인
